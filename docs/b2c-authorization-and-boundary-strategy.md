# B2C向け 認可・データ境界戦略 (B2C Authorization & Boundary Strategy)

## 1. 背景と目的
当初はB2B SaaS向けの「高度なPolicy Engine（JSONベース）」や「厳密なTenant/Organization分離」を検討していましたが、本IdPの主目的が **B2C（コンシューマー向け）基盤** であることにフォーカスし、過剰な複雑性を排除します。

B2Cにおいて最も重要な「スケーラビリティ」「軽快なUX（摩擦のない認証）」「運用保守のシンプルさ」を最大化するため、認可とデータ境界の基本方針を再定義します。

## 2. データ境界（厳格なテナント分離の廃止）
B2Cでは、ユーザーは企業（テナント）に属する従業員ではなく、独立した「個のコンシューマー」です。

- **フラットなユーザープール:**
  すべてのユーザーは単一の `users` テーブルでフラットに管理され、`tenant_id` のような物理的/論理的な厳しいデータの壁（マルチテナント境界）は設けない方針とします。
- **Organization / Group の扱い:**
  現在のDBスキーマに存在する `organizations` や `groups` は、「セキュリティの絶対的な境界」としてではなく、「ファミリーアカウント」や「招待グループ」などの **任意かつ緩やかなユーザーのグルーピング機能** としてのみ利用します（無理にIdPのコア機能に組み込まない）。
- **もたらされるメリット:**
  複雑なクロステナントのバグや、全クエリに対する `tenant_id` によるフィルタリング義務がなくなり、DBスキーマとRepository層が極めてシンプルに保たれます。

## 3. 認可モデル（複雑なPolicy Engineの廃止）
独自のJSONベースのPolicy Engine（ルール評価エンジン）は自作せず、標準的でシンプル、かつデバッグのしやすい手法に留めます。

- **Scope と Claim ベースの認可 (委譲):**
  IdPの主責務は「認証」と「標準トークンの発行」に特化します。ユーザーの属性や基本的な権限は OIDCの `scopes` や ID Token の `claims` に乗せてクライアント（各アプリケーションやAPI Gateway）へ渡します。**「特定の数量制限」や「休日・時間帯によるアクセス制限」などの複雑なビジネスルールは、IdP内部ではなくアプリケーション側のロジックとして評価・制御** させます。
- **シンプルな Role ベース認可 (Basic RBAC):**
  IdP自体の管理画面（カスタマーサポート用など）へのアクセス制御のため、`admin` / `support` / `user` のような最低限のRoleのみをコードベースで付与・判定します。
- **コードベースの所有権チェック (Ownership Check):**
  「自分自身のプロフィールしか編集できない」といったAPIレベルでの認可は、MiddlewareやService層での単純な `if (resource.ownerId !== currentUser.id)` といった明示的なTypeScriptコードのガード句で処理し、可読性を優先します。

## 4. リソースの再配分（B2C特有の重要課題への集中）
複雑なB2B要件を削った分の開発リソースは、以下の **B2C基盤として勝敗を分けるコア機能** に集中的に投資します。

1. **フリクションレスな認証体験 (Frictionless UX):**
   - Passkeys (WebAuthn) によるパスワードレス認証の洗練と定着
   - 各種ソーシャルログイン (Google, Apple, LINE等) のシームレスな統合と、安全なアカウントリンク機構
2. **パフォーマンスと高可用性:**
   - 突発的なトラフィック急増（テレビ放送やキャンペーン時のスパイク）に耐えうるキャッシュ戦略やDBチューニング
3. **コンシューマー向けセキュリティ機能 (Security Monitoring):**
   - Credential Stuffing（リスト型アカウントハッキング）を防ぐための強力なレートリミットと異常検知（09番の計画へ注力）
   - アカウントの安全なリカバリーフロー（MFA紛失時のセルフサービス復旧など、サポートコストを下げる仕組み）

## 5. 結論
B2C基盤においては、「IdPがいかに複雑な認可ルールを解釈できるか」よりも**「いかに速く、安全に、そして摩擦なくユーザーをログインさせてサービスに送り出せるか」**が最大の価値となります。

今後は **「KISSの原則 (Keep It Simple, Stupid)」** に従い、IdP本体に過剰なビジネスロジックを抱え込ませず、シンプルで高パフォーマンスな堅牢認証プロバイダーとして磨き上げていきます。
